# 双色球历史档案单文件页面：今日更新说明

覆盖范围：今天围绕单文件 HTML 工具所做的全部产品、算法、UI、性能与稳定性更新

## 一、总览

今天的工作不是一次小修补，而是围绕同一个单文件 HTML 工具做了连续多轮迭代：先完成基础版，再加入推荐与评分，再重构为适合个人使用的推荐器，再引入“评分带形态系统”，最后做体积优化与逻辑审查。

整体上，今天的更新可以拆为五条主线：

1. 功能补齐
2. 推荐逻辑重构
3. 评分体系修正
4. 体积优化
5. 代码稳定性修复

最终结果不是若干分散原型，而是一个连续演进到 v8 的单文件版本链。

---

## 二、版本演进明细

### 1）初版单文件（v1）
- 把设计文档与 CSV 历史数据整合进一个 HTML。
- 数据、JavaScript、样式全部内嵌。
- 实现基础历史档案、形态统计、查询与展示。

**定位：**
- 做一个不依赖后端、不依赖外链、可本地双击打开的验证原型。
- 排除“爬数据”和“数据清洗”链路，只保留页面端可完成的历史展示、形态识别、档案化浏览。

### 2）实验区版（v2）
- 在主页面中增加单独的验证区域。
- 加入输入号码评分与实验性推荐。

**特点：**
- 仍保留“未来可能发布时把推荐区整体删除”的思路。
- 推荐逻辑采用“随机采样 + 结构评分 + 规则过滤”，而不是传统冷热号叙事。

### 3）个人使用重构版（v3）
- 放弃“未来发布要删掉实验区”的约束，直接按个人长期使用工具重构。
- 新增“本期推荐”主功能：每次推荐 5 组，可勾选排除特殊形态，可排序，并能展开查看每组形态分析。
- 最近开奖默认只显示最近一期，点击后再展开更多。
- 页面从大块平铺式 div 重构为“列表 + 折叠详情”的交互结构。

### 4）历史当期评分版（v4）
- 在历史开奖详情中增加“截止当时”的综合评分、常见度评分、均衡度评分。
- 单独整理公式说明文档。

**意义：**
- 用户可以看到某一期号码若站在当时视角，会被当前系统打成什么分。
- 同时补了一份文档，解释综合评分、常见度、均衡度、AC、和值、跨度、连号、尾数等名词与计算口径。

### 5）评分带初步版（v5）
- 把综合评分、常见度、均衡度各自映射到“评分带”。
- 在页面中展示评分带及其历史占比。

**意义：**
- 弱化“高分最好”的直觉。
- 承认“分数落在哪个区间”本身也是一个可观察维度。

**局限：**
- 这一版仍主要停留在展示层。
- 推荐逻辑还没有完全切换到评分带密度思路。

### 6）密集带重构版（v6）
- 正式把“评分带形态系统”做实。
- 历史开奖按**不含当期自身**的口径回算当期分数。
- 据此建立三套评分带档案：综合评分带、常见度带、均衡度带。
- 推荐逻辑改成优先贴近历史高密集评分带。

**这是今天最重要的一次逻辑转向。**

具体表现为：
- 综合评分不再被解释为“越高越好”，而是被解释为一种聚合后结果带。
- 推荐排序从单纯看分值，转为“结构分 + 评分带密度贴合度”。
- 常见度与均衡度被提升为更重要的主分布轴；综合评分则保留为辅助投影轴。

### 7）性能优化版（v7）
- 在不削减功能的前提下，把单文件体积从约 8.1MB 压到约 661KB。

**结论：**
- 问题根源不是样式，而是旧版把大量预计算 JSON 直接展开写入 HTML。
- 本次通过压缩和运行时恢复，把庞大的预计算载荷收缩，大幅降低页面体积。

### 8）检查修正版（v8）
- 对现有所有代码做逻辑检查。
- 修复双重渲染和全局访问稳定性问题。

**修复内容：**
- 去掉旧逻辑和新逻辑先后各渲染一次的结构，避免旧口径首屏渲染后再被新版覆盖。
- 把内联 `onclick` 需要访问的 `state`、`draws` 和关键函数显式挂到全局对象上，降低浏览器兼容风险。
- 对推荐、历史评分、评分带档案的总量一致性进行验证。

## 三、今天最重要的逻辑变化

### 1）从“高分更好”转为“评分带也是形态”
最初的综合评分更像一个单轴结果，容易被误读成越高越值得推荐。今天最终明确把综合评分改为“结果带标签”，即关注它落入哪一个历史分数区间，而不是只关注数值高低。

这意味着：
- 历史开奖不是简单看哪个号码评分高。
- 而是统计每期开奖在当时历史截面下落入了哪个评分带。
- 再分析哪些评分带是主密集带、哪些是次密集带、哪些是低密度带。

这个思想随后扩展到另外两条评分轴：
- 历史常见度
- 结构均衡度

于是页面中的评分系统从单点分数展示，升级为三套“评分带形态档案”。

### 2）推荐逻辑从“高分排序”转为“贴近历史高密集评分带”
推荐器不再单纯追求综合评分最大，而是：

1. 先用结构规则完成过滤。
2. 再把候选号码放回历史评分带分布中查看。
3. 优先挑选更贴近历史高密集评分带的组合。

因此，推荐排序分本质上是：
- 结构基础分
- 加上评分带密度贴合度

其中：
- 常见度带和均衡度带权重更高
- 综合评分带只作为聚合后的辅助投影轴

### 3）“截止当时评分”口径纠偏
在中期实现中，历史开奖“截止当时”的评分存在轻微数据泄漏：当期开奖本身被算进了自己的参考历史里。

后续版本把这一点修正为：
- 当期号码的三项评分只允许使用更早之前的历史数据进行计算
- 不得包含当期自身

这个修正非常关键，否则评分分布会被轻微污染，尤其在早期开奖样本较少时更明显。

## 四、UI 与交互更新说明

- 页面交互从平铺式大块展示，调整为“列表为主、详情折叠”的结构。
- 默认状态尽量只显示标题行、核心摘要和少量最新数据，避免首次打开信息过载。
- 最近开奖默认只显示最近一期；查看更多后再逐步展开。
- 形态档案、公式说明、推荐结果分析、历史开奖详情都尽量收拢为折叠区块。

**目标不是更花哨，而是减少一次性渲染与浏览负担，让用户把注意力放到需要看的那一项上。**

## 五、性能优化说明

| 维度 | 优化前 | 优化后 / 结论 |
|---|---|---|
| 文件体积 | 约 8.1MB | 约 661KB，下降约 92.2% |
| 主要问题 | 把大量预计算 JSON 直接写入 HTML | 改为压缩载入与运行时恢复 |
| 样式影响 | 不是主因 | 即便极限简化 CSS，也无法带来同量级缩减 |
| 功能影响 | 体积大、加载与维护成本高 | 功能保持不变，仅收缩数据载荷 |

这一步的关键判断是：
**页面大不是因为 UI，而是因为数据膨胀。只压 CSS 没意义，真正需要处理的是巨大的预计算中间结果。**

## 六、逻辑检查与修复说明

### 1）发现的问题
- 同一份 HTML 中保留了旧版与新版两套实现，旧版会先渲染一次，之后新版再覆盖；这会造成首屏口径不一致和后续维护混乱。
- 大量按钮使用内联 `onclick`，而相关的状态对象与函数并未明确挂到全局对象上；部分浏览器虽然可运行，但属于不稳的隐患。

### 2）采取的修复
- 移除首次旧渲染，保留最终生效的新版渲染链路。
- 把 `state`、`draws`、`latestDraw` 以及核心交互函数显式挂到全局对象，保证内联事件处理器能稳定访问。
- 对推荐、历史评分、评分带档案的总量一致性进行验证。

### 3）当前判断
v8 未发现会直接导致主功能算错的显性逻辑错误。当前剩余风险更多属于“模型与规则是否符合长期使用偏好”的产品层问题，而不是代码级立即故障。

## 八、当前建议的使用基线

建议你后续验证一律以 `v8` 为基线，因为它同时包含了：
- 评分带形态系统
- 历史“截止当时评分”口径修正
- 推荐器的高密集评分带贴合逻辑
- 压缩后的载荷
- 运行稳定性修复

如果后面继续迭代，优先级建议不是再堆功能，而是做三类事情：

1. 校验推荐排序分的权重是否符合你的使用习惯。
2. 检查早期开奖样本较少阶段的评分波动是否需要特殊解释。
3. 把评分带系统再做一份正式规则文档，便于后续维护与迁移。

## 九、可继续深化但今天未处理的点

- 推荐器的权重仍属启发式规则，今天的重点是逻辑转向与可用化，不是统计学严谨建模。
- 评分带如何分箱、每箱宽度如何设定、是否需要动态分位数分箱，这些今天没有再展开。
- 页面仍是单文件实现，后续若要长期维护，最好把“数据、计算、UI、文档说明”拆成更清晰的模块化源码，再生成单文件发布物。
